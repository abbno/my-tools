# SSH 密钥自动设置 - CMD 窗口交互方案设计

## 背景

SSH 密码认证在后台进程中无法工作，需要用户交互式输入密码。之前的 PowerShell SendKeys 方案因窗口激活失败无法可靠工作。

## 解决方案

改用弹出独立 cmd 窗口的方式，让用户在其中交互式输入 SSH 密码，部署完成后自动检测验证。

## 用户交互流程

```
用户点击"保存"或"测试"
    ↓
前端弹出密钥设置对话框
    ↓
后端生成/检查本地 SSH 密钥 (~/.ssh/id_ed25519)
    ↓
弹出 cmd 窗口执行 SSH 命令部署公钥
    ↓
用户在 cmd 窗口交互输入密码
    ↓
用户关闭 cmd 窗口
    ↓
后端自动检测：尝试用密钥 SSH 连接验证
    ↓
成功 → 返回成功结果，前端显示"密钥设置成功"，保存配置
失败 → 返回失败结果，前端显示错误，提供"重试"按钮
```

## 技术实现

### 1. 后端密钥生成（保持不变）

- 使用 ssh-key crate 生成 Ed25519 密钥
- 保存到 ~/.ssh/id_ed25519 和 ~/.ssh/id_ed25519.pub

### 2. CMD 窗口公钥部署（新方案）

**Windows 实现:**

```rust
// 使用 Command 打开 cmd 窗口
Command::new("cmd")
    .args(["/C", "start", "wait", "cmd", "/K"])
    .arg(ssh_command)
    .status()  // 等待窗口关闭
```

SSH 命令格式:
```
ssh -o StrictHostKeyChecking=accept-new -p <port> <user>@<host> "mkdir -p ~/.ssh; chmod 700 ~/.ssh; echo '<公钥>' >> ~/.ssh/authorized_keys; chmod 600 ~/.ssh/authorized_keys"
```

用户在 cmd 窗口中看到密码提示，输入密码后命令执行。

### 3. 自动检测验证

执行批量模式 SSH 连接验证密钥是否部署成功:

```rust
Command::new("ssh")
    .args(["-i", key_path])
    .args(["-o", "BatchMode=yes"])
    .args(["-o", "StrictHostKeyChecking=accept-new"])
    .args(["-p", port])
    .arg(format!("{}@{}", username, host))
    .arg("exit")
    .status()
```

- 退出码 0 = 成功
- 退出码非 0 = 失败（可能是密码错误、网络问题等）

### 4. 重试机制

失败时前端显示错误信息，用户可点击"重试"按钮重新弹出 cmd 窗口。

## 前端界面变化

### 配置表单 (ConfigForm.vue)

移除以下字段:
- 认证类型选择（authType）
- 密码输入框
- 密钥路径输入框
- 密钥密码输入框

保留字段:
- 配置名称、所属分组
- 主机地址、端口、用户名
- 隧道配置（类型、本地/远程地址和端口）
- 高级选项（常用、开机启动、自动重连）

### 密钥设置对话框 (KeySetupDialog.vue)

状态流转:
1. **等待状态**: 显示"请在弹出的命令窗口中输入密码..."
2. **检测状态**: cmd 窗口关闭后显示"正在验证密钥部署..."
3. **成功状态**: 显示"密钥设置成功"，密钥路径，"完成"按钮
4. **失败状态**: 显示错误信息，"重试"按钮，"取消"按钮

## 错误处理

| 场景 | 处理 |
|------|------|
| 密码错误 | cmd 窗口显示 SSH 错误，用户可多次输入，关闭后自动检测失败，提示重试 |
| 网络问题 | 自动检测失败，提示"连接超时"或"网络错误" |
| 主机密钥验证失败 | 提示具体错误，允许重试 |
| 密钥生成失败 | 直接返回错误，不弹出 cmd 窗口 |

## 优势

1. **可靠性**: 用户直接交互，无需模拟键盘输入
2. **简单性**: 利用系统原生 SSH，无需额外工具
3. **透明性**: 用户能看到 SSH 输出，了解部署状态
